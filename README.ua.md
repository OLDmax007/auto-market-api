# AutoMarket API

[Клікніть для читання на англійській](./README.md)

Просунута платформа для продажу автомобілів, розроблена з фокусом на високу продуктивність, гнучкість архітектури та можливість майбутнього масштабування до рівня великих автосалонів.

## Архітектура та Технічні рішення

Згідно з тз розроблено:

* **Hybrid RBAC/ABAC System**: Доступ побудований на системі пермішинів (Permissions). Це дозволяє легко додавати нові ролі (наприклад, "Механік" або "Сейл для автосалону") без зміни базового коду.
* **Scalability Ready**: Модульна структура backend-частини дозволяє виносити окремі сервіси (наприклад, модерацію або аналітику курсів) у мікросервіси в майбутньому.
* **Automatic Moderation**: Інтегрована система перевірки контенту на нецензурну лексику.
* **Dynamic Currency Engine**: Розрахунок цін у реальному часі на основі API ПриватБанку з фіксацією курсу на момент публікації.

## Технічні особливості та інтеграції

### 1. Робота з медіа-файлами (AWS S3)
* **Масштабованість**: Використання хмарного сховища дозволяє системі обробляти велику кількість контенту без навантаження на локальний диск сервера.
* **Оптимізація**: Валідація форматів (JPG, PNG) та обмеження розміру файлів на рівні Middleware.
* **Безпека**: Контроль доступу до видалення об'єктів — видаляти фото можуть лише автори оголошень або модератори.

### 2. Система сповіщень (Nodemailer)

* **Верифікація акаунта**: Безпечне підтвердження електронної пошти після реєстрації.
* **Support Tickets**: Автоматична генерація та відправка запитів адміністрації, якщо користувач хоче додати нову марку або модель авто в систему.

### 3. Обробка та валідація даних
* **Mongoose Middleware**: Автоматичне хешування паролів (bcrypt) та глибока валідація даних перед збереженням у MongoDB.
* **Продуктивність (Indexing)**: Налаштовані бази індексів для найбільш часто використовуваних полів пошуку (марка, ціна, регіон), що гарантує швидку роботу фільтрів при великих обсягах даних.
* **Стратегія нормалізації даних**:
    - **Технічні ключі**: Марки, моделі та регіони зберігаються виключно у нижньому регістрі (`lowercase`). Це гарантує високу швидкість індексації та стабільну роботу фільтрів при будь-яких запитах.
    - **Дані для відображення**: Назви міст зберігаються у форматі `Title Case` (наприклад, "Bila Tserkva"), щоб зберегти правильне написання для UI без необхідності додаткової обробки на фронтенді.
    - **Користувацькі пропозиції**: Система дозволяє пропонувати нові марки або моделі через спеціальний шар валідації (на базі Regex). Такі записи автоматично позначаються для подальшої перевірки адміністратором.

### 4. Автоматизація (Cron Jobs)

* **Курси валют**: Щоденне автоматичне оновлення курсів (USD, EUR) через API ПриватБанку для актуалізації цін в оголошеннях.
* **Керування підписками**: Автоматична перевірка термінів дії преміум-акаунтів та їх переведення на базовий тариф після завершення терміну.

### 5. Розширений пошук та фільтрація (Query Engine)
* **Гнучка фільтрація**: Підтримка складних параметрів запиту, включаючи діапазони цін (`minPrice`, `maxPrice`), рік випуску та технічні характеристики автомобіля.
* **Оптимізація пошуку**: Автоматична нормалізація пошукових запитів. Система підтримує регістронезалежний пошук (case-insensitive) для марок та моделей авто, що забезпечує стабільну роботу фільтрів незалежно від формату введення.
* **Пагінація та сортування**: Вбудована підтримка параметрів `limit`, `offset` та динамічного сортування (за ціною, датою публікації або популярністю). Це дозволяє оптимізувати продуктивність фронтенду та зменшити обсяг переданих даних.

### 6. Початкове наповнення бази (Database Seeding)

Щоб платформа була готова до роботи відразу після розгортання, вбудований сідер автоматично виконує наступні дії:

* **Каталог автомобілів**: Наповнює базу даних детальною картою марок та моделей (`BMW`, `Daewoo` тощо) для роботи випадаючих списків та фільтрів.
* **Географія**: Попередньо заповнює регіони та основні міста для стандартизації пошуку за локацією та забезпечення цілісності даних.
* **Ініціалізація RBAC**: Створює структуру основних ролей (`admin`, `manager`, `seller`, `visitor`) та закріплює за кожною роллю відповідний набір пермішинів (прав доступу).
* **Системні дозволи**: Ініціалізує всі необхідні права доступу, які регулюють взаємодію з конкретними API-ендпоінтами.

## Ролі в системі

| Роль | Опис |
| :--- | :--- |
| **Покупець** | Базова роль для пошуку авто та зв'язку з продавцями. |
| **Продавець** | Може розміщувати оголошення (Базовий: 1 авто, Преміум: безліміт). |
| **Менеджер** | Модератор: перевіряє оголошення, банить порушників. |
| **Адміністратор** | Повний доступ до системи та управління персоналом (менеджерами). |

## Технологічний стек

* **Backend**: Node.js, TypeScript, Express.
* **Database**: MongoDB (Mongoose).
* **Documentation**: Swagger (OpenAPI 3.0).
* **DevOps**: Docker, Docker Compose.

# Запуск

## 1. Встановлення та оточення

Встановіть залежності локально. Це потрібно для роботи npm-скриптів та підсвітки коду в IDE:

```bash
npm install
```

Для стабільної роботи платформи необхідно налаштувати змінні оточення.

У проекті вже є приклад файлу змінних — **`.env.example`**.  
Створювати `.env` вручну не потрібно. Просто скопіюйте приклад:

```bash
cp .env.example .env
```

### Деталі налаштування оточення

Вам необхідно налаштувати файл **`.env`** для коректної роботи наступних сервісів:

* **Автентифікація**: Згенеруйте власні унікальні рандомні рядки для всіх змінних `JWT_SECRET`.
* **База даних**: Вкажіть ваш `MONGO_URI`. Система підтримує як MongoDB Atlas, так і локальні інстанси.
* **Електронна пошта (Nodemailer)**: Використовуйте **"Пароль додатка"** (App Password) для Gmail у змінній `EMAIL_PASSWORD`. Це необхідно для верифікації акаунтів.
* **Хмарне сховище (AWS S3)**: Використовується для зберігання фото автомобілів. Переконайтеся, що ваш IAM-користувач має права **`PutObject`** та **`DeleteObject`**.
* **URL додатка**:
    * `FRONTEND_URL`: Необхідний для генерації валідних посилань у листах верифікації (через `buildLink`). Ваш фронтенд обробляє токен і викликає API.
    * `BACKEND_URL`: Базовий URL вашого API, що використовується для внутрішньої логіки та налаштування CORS.


## 2. Налаштування супер адміна

Для автоматичного створення адміністратора та наповнення бази тестовими даними використовується вбудований сідер.

### Конфігурація через `.env`
Усі параметри супер-адміна підтягуються зі змінних оточення. Ви можете змінити їх у файлі `.env` перед запуском:

* **Email**: `SEED_ADMIN_EMAIL` — використовується для логіну.
* **Password**: `SEED_ADMIN_PASSWORD` — пароль для першого входу.
* **Персональні дані**: `SEED_ADMIN_FIRSTNAME`, `SEED_ADMIN_LASTNAME`, `SEED_ADMIN_AGE`.

### Тестові можливості
Для зручності тестування в скриптах сідера передбачено:

* **Початковий баланс**: Супер-адміну встановлюється тестовий баланс у розмірі **100 000 грн**. Це дозволяє миттєво перевірити купівлю **Premium** акаунтів та інших платних послуг без необхідності ручного поповнення.
* **Ролі**: Автоматичне призначення ролі `ADMIN` з повним набором прав (пермішинів) для супер-користувача.
*
**Примітка:** Якщо потрібно оновити дані адміна або баланс після того, як база вже була ініціалізована, виконайте команду:
```bash 
docker-compose run --rm seeder 
```

## 3. Опис Docker сервісів та скриптів

Проєкт повністю контейнеризований і складається з двох основних сервісів. Весь конфіг базується на образі `node:20-alpine`.

### 1. Сервіси

| Сервіс | Контейнер | Порт | Опис |
| :--- | :--- | :--- | :--- |
| **backend** | `auto-market-backend` | `5000:5000` | Основний API додаток (NestJS/Express). |
| **seeder** | (Тимчасовий) | - | Скрипт для ініціалізації бази даних. |

### 2. Скрипти та команди

Для зручності в `package.json` додано скорочення для основних команд Docker Compose. Це дозволяє не вводити довгі команди вручну:

| Команда                     | Що робить                              | Docker еквівалент                  |
|-----------------------------|---------------------------------------|-----------------------------------|
| `npm run docker:up-all`     | Запуск всіх сервісів (API + Seeder)  | `docker-compose up --build`       |
| `npm run docker:up-backend` | Запуск тільки API                     | `docker-compose up backend`       |
| `npm run docker:run-seeder` | Наповнення БД (тимчасовий контейнер) | `docker-compose run --rm seeder` |
| `npm run docker:down`       | Зупинка всіх контейнерів              | `docker-compose down`             |

### 3. Технічні деталі (Dockerfile)
* **Робоча директорія**: `/app`
* **Залежності**: Використовується `tsx` для прямого запуску TypeScript файлів без попередньої компіляції в JS (ідеально для дев-середовища).
* **Volumes**: Ваші локальні зміни в коді (`./backend`) миттєво синхронізуються з контейнером завдяки монтуванню волюмів.

# Тестування через Postman

У папці `postman/` знаходяться файли колекції та оточення. Колекція побудована за принципом **End-to-End Flow**, що дозволяє протестувати весь цикл роботи додатка без ручного копіювання ідентифікаторів.

## 1. Підготовка до роботи
* **Імпорт**: Завантажте файл `collection.json` (запити) та `environment.json` (змінні) у Postman.
* **Оточення**: Обов'язково оберіть environment **"auto-market"** у верхньому правому куті Postman.
* **Змінна HOST**: Переконайтеся, що значення `{{HOST}}` відповідає вашому локальному порту (за замовчуванням `http://localhost:5000`).

## 2. Автоматизований сценарій (Flow)
Запити в колекції розташовані в логічній послідовності. Завдяки вбудованим скриптам, дані передаються між запитами автоматично:

1.  **Авторизація (`Auth`)**: Виконайте запит `Sign Up` або `Sign In`. Скрипт автоматично збереже **`accessToken`**, **`refreshToken`** та ваш **`TEST_USER_ID`** у змінні оточення.
2.  **Створення оголошення (`Listings`)**: Після виконання `Create Listing`, `_id` вашого нового оголошення автоматично запишеться в змінну **`TEST_LISTING_ID`**.
3.  **Дії з об'єктами**: Наступні запити (Get by ID, Update, Delete) вже використовують ці змінні, тому працюють миттєво без ручного введення.
4.  **Action Токени (`TEST_ACTION_TOKEN`)**: Для верифікації Email або скидання пароля токен надсилається зовнішнім каналом:
    * **Де взяти**: Скопіюйте рядок токена з листа на пошті або з урли (частина після `token=...`) або з консолі сервера.
    * **Куди вставити**: Вставте отримане значення в змінну `TEST_ACTION_TOKEN` в налаштуваннях оточення Postman.

## 3. Структура та Модулі

Колекція організована таким чином, щоб розділити публічні операції, дії користувача та адміністративні функції:

* **🔐 Auth**: Керування доступом.
    * Реєстрація, вхід, верифікація через пошту.
    * Скидання пароля та механізм `refresh token`.
    * Можливість вийти з усіх пристроїв одночасно (`logout all`).
* **🚗 Car**: Робота з базою авто. Отримання списку марок та моделей, доступних у системі та відправка листа про не існуючу модель або марку.
* **📋 Listings**: Робота з оголошеннями.
    * Створення, редагування, перегляд статистики оголошення та пошук авто.
    * 📁 **Staff (вкладена)**: Модерація оголошень та керування будь-яким лістингом.
* **👤 Me**: Особистий кабінет.
    * Керування балансом (`top up balance`) та зміна статусу на продавця.
    * Купівля преміум-плана (`upgrade plan`).
    * Завантаження або видалення аватарки
* **👥 Users**
    * Перегляд публічної інформації юзера.
    * 📁 **Staff  (вкладена)**: Модуль для адміністраторів.
        * Перегляд списку всіх користувачів.
        * Керування ролями та статусами користувачів.
        * Можливість видалення або блокування акаунтів.
* **💎 Subscriptions (Staff)**: Адмін-панель.
    * Змінна підписки.
    * Активація/Деактивація підписки.
